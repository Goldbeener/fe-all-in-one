在LLM大模型的基础上，扩展了
1. 上下文召回， 收集历史信息，参与决策
2. 流程编排， 决定执行顺序
3. 工具调用，获取外部数据

普通的LLM应用强调生成回复，单次的对话，无状态，上下文关联没有或不强
AI Agent应用强调为了完成目标而组织动作

RAG 检索增强生成，持久记忆

**数据沉淀**，根据对话信息，提取关键信息，存入外部记忆库

**向量化索引**，将关键信息向量化，转化成一组数字存入向量数据库，语意相近的文本数字相近

**语意检索**，新消息也转换成向量，与向量数据库内容对比，找到相近的历史记录，召回相关信息

**上下文组装**， 召回信息，当前信息、最近几次对话信息，拼装成完成提示词，传递给大模型

**模型生成**，大模型根据增强的提示词，生成回复

**记忆更新**，根据本次对话，产生新的记忆，更新
	记忆方案： 根据具体场景，用户量/记忆量
		单文件存储
		SQLite + 向量混合检索

 Agent 更像一个有输入、有状态、有动作、有输出的个体，

单个Agent 全能助手，任务拆分、工具调用全部负责

多Agent，最重要的是分工，不同的agent负责不同的任务，agent之间相互协调调用，最终输出结果

LangChain 负责把单个Agent + 工具调用 单元搭建起来
LangGraph 承接多个Agent的协调、状态流程、多步骤流程， 更多是调度职责


LangChain
+ 模型接入
+ 输入组织
+ 链路编排
+ 工具接入
+ 一轮请求处理
+ 记忆和检索接入

组装出一个能真正独立工作的一个Agent单元



简单情况下，单Agent就可以胜任， 依靠LangChain
随着Agent的能力扩展，依靠单个Agent完成任务会不好维护、不好排查、效率不高，就需要考虑转向多Agent

多个Agent 最重要的还是**分工协作**，以及**隔离上下文**
每个子agent在干净的窗口工作，不被其他任务信息污染，减少token消耗

这时候需要LangGraph居中调度


LangGraph 解决不是模型，而是**流程**
先解决的事流程编排的问题，其次才是多Agent问题

问题不是一步解决的，要解决问题需要拆分成多个步骤，每一步的结果会相互影响、产生不同分支，最终汇总得到结果。
LangGraph就是维护这个过程中的状态，以及根据状态做流程编排


LangGraph
+ 组织流程
+ 保存状态
+ 处理分支
+ 处理回调
+ 多节点协作
+ checkPoint
	+ 节点恢复


一个要点是：
**Claude Code 的多 agent 本质上是编排模式**，
主 agent 负责拆分任务、派发给子 agent，
但默认情况下主 agent 是**逐个等待子 agent 完成后再派发下一个**，而不是同时启动所有子 agent。

few-shot
是提示工程（PromptEngineering）中的核心概念，指在给模型的提示中提供少量示例，让模型通过这些示例"学会"如何完成任务

+ 示例要最小够用
+ 注意不要与系统设定冲突
+ 边界案例优先，不需要给正常情况的示例，要给容易出错的场景
