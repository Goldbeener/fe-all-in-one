
# 内存调度
大模型是无状态的，每次请求对于大模型来说都是一个全新的、独立的请求
不同的请求之间是没有关联关系的

> 类似于http的无状态

但是对于agent这种场景来说是不能接受的
因为agent的请求是连续的，有关联的

但是又不能把所有信息都丢进prompt中，那样会导致上下文爆炸，浪费token

因此需要一个调度层，决定哪些信息要进prompt、哪些要压缩，哪些要长期存储，哪些要丢弃

这就是**内存调度器**

+ 长期记忆的精准调用，需要从历史会话中获取关键信息
+ 场景/情绪切换识别
+ 多模态感知与工具调用

调度器主要工作，根据任务优先级，确定任务的执行顺序

以及一些边界问题处理


LangGraph 和 LangChain


LangGraph似乎就是任务的分发角色，识别、规划出为了回答问题，需要做什么事情
LangChain就是具体干活的



# 向量化与语义检索

存储持久化信息，
用于记录一些用户基础信息、偏好信息、历史选择等，
在特定时机注入上下文，方便大模型做更合适的回答

关系到关键信息的提取、存储、读取

数据源，来自于用户的历史每轮对话，拆分提取出关键信息

存储形式/检索形式
这两者是相关的，从数据使用的期望来说，对这些信息需要做的是语义检索
不是普通的文本匹配，不同的文本表达含义一样，那就希望被检索、关联到
所以，存储/检索 采用了向量化存储，方便做语义检索
因为，含义相同的话，向量化之后空间位置相近

> 使用余弦相似度衡量语义相近
> 使用vectorize存储


检索策略优化
+ 控制切分粒度
+ 限定条件+向量匹配
+ 重排序
+ 
做这些事情的**最终目的**是： **丰富提示词**
用户在单次对话中，输入较少的信息，就可以用过向量检索，提取出期望信息，再检索召回历史得到的用户偏好，丰富提示词，以便大模型能够有足够多的上下文信息，做出准确度较高的回答
