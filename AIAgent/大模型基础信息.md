
大模型
LLM的本质是Token预测

BPE： Byte Pair Encoding

模型训练之前，先统计大量文本中哪些**字符组合**出现频率最高，把高频组合合并成一个token
反复迭代之后，形成一个固定大小的词表

**token是语言统计规律的产物**

英文环境下，
简单、高频的单词可能就直接是一个token
复杂、生僻的单词拆分成多个token

中文环境下
每个汉字通常对应一个token
部分高频词组会合并， 但因为中文组合可能性太多，只能将完全没有歧义、准确的词组组合成token，更多的还是单个汉字一个token

**所以语义相同的一段话，英文的往往消耗的token更少。**
因为可能一句话里面，某个token出现了好几次，
但是中文中一个汉字就是一个token


> 如何理解大模型中的token，这个token的产生和应用具体是怎么样的？
> token与自然语言的对应关系是什么样的


LLM 回答问题，是自回归生成
就是先生成一个token，然后把第一个token加入输入，传递给LLM
再进行第二轮输出，产生第二个token
...
直到回答完毕

**输出是一个token一个token往外蹦的**

核心特性就是，**下一个token依赖所有的前面的token**


> 但是输入场景下token是已经确定了的，可以在transformer层并行处理
> 输出场景下，token产生机制，导致了有几个token就需要transformer层处理几次， 每处理一次就需要一次算力成本
> 因此**输出计费往往比输入更贵**


这种机制导致了
1. 大模型输出天然是流式输出
2. 工具的调用，需要攒够完整信息才能解析
3. response prefill 能够操控模型的行为
	1. 拦截响应，在前面加上固定信息，就能够限制模型在指定域内产生

# 模型层
**Context Window** 上下文窗口，模型一次能看到最大的token数，决定大模型能处理多大的任务，目前顶级的模型已经到200k甚至1Mtoken

**Multimodal** 多模态，大模型是否能处理文本之外的信息，比如处理图片、音视频等

**Reasoning Modal** 推理模型， 强化慢思考能力的模型，在复杂数学、逻辑规划类任务上优于普通模型

# 架构与工程层

**RAG** 检索增强生成，解决大模型不知道私有数据的问题

**Vector Database** 向量数据库，RAG的基础设施，把文本转换成向量存储，支持语义相似度检索

**Embedding** 嵌入， 把文本、图片转换成向量的过层，是RAG和语义搜索的底层技术

**Fine-tuning** 微调 ，在通用模型基础上使用特定数据继续训练，使模型能更好的解决特定领域问题

**Prompt Engineering** 提示工程， 设计和优化输入给大模型的提示词，直接影响输出质量， skills本质上就是一种prompt engineering的实践

# Agnet与自动化层
**Multi-Agent** 多智能体，多个智能体协作完成任务，各自不同分工

Orchestrator/Planner 编排器，在多智能体场景中，负责任务分配和协调的角色

Memory 记忆，智能体的记忆管理机制，分短期（当前对话上下文）、长期（持久化存储）、语义记忆（向量检索）三种，解决大模型无状态的根本问题

Tool Use/Function Calling 大模型调用外部工具的标准机制，MCP是上层协议




AI Agent
+ 自主规划
+ 工具调用
+ 反馈循环


Agent 是对大模型的一层包装， 是以应用的形式使用大模型
里面有几方面主要概念

**commands**： 减少重复输入，以快捷键的形式，实现特定任务，本质是预设的prompt模板

**MCP/Tools**： MCP 工具扩展协议，让大模型可以调用外部服务，扩展大模型的能力，有时候大模型需要一些外部数据来更好的回答问题，就可以通过MCP获取

**skills**： 领域知识最佳实践手册，指导大模型按照最佳实践的步骤处理问题

**hooks**： 自动触发器，事件驱动的钩子机制，在特定时机自动触发预设动作，解决的是自动化流程编排问题

**plugins**： 对智能体本身能力的扩展包，可以改变智能体的行为方式，而不是增加它能调用的工具（MCP干的事）

**Agent SDK** 把智能体的能力嵌入到自己的应用或工作流中
